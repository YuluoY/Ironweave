# 结构化选项与推荐项规则

## 选项设计规则

- 每个问题提供 **2-5 个预设选项**
- **必须且只能标记一个推荐选项**，并在选项描述中简要说明推荐理由
- **最后一个选项**始终为开放式输入项（如"其他：请描述你的想法"），允许自由输入
- 涉及多维度选择时允许多选
- 问题前简要说明"为什么现在问这个"——让用户理解提问的意义

## 推荐项判断依据（按优先级）

1. **工程事实**：当前仓库/项目已有的技术栈、架构模式、配置约定
2. **上下文推断**：从之前轮次收集的已确认事实推断最可能的选择
3. **行业惯例**：当工程事实和上下文不充分时，选择该领域最主流的方案
4. **安全侧偏好**：多个选项风险差异大时，推荐风险最低的方案

## 推荐项在不同宿主中的表达

| 宿主能力 | 推荐项表达方式 |
|---|---|
| 支持 `recommended` 字段（如 VS Code `askQuestions`） | 直接标记 `recommended: true` |
| 支持选项描述 | 在推荐项前加 ⭐ 或描述中写"推荐" |
| 仅纯文本选项 | 推荐项放第一位，文本标注"（推荐）" |
| 退回聊天消息 | 用加粗标记推荐项 |

无论宿主能力如何，推荐项的**判断逻辑不变**，只有表达方式随宿主适配。

## 禁止

- 不标记任何推荐项（增加用户决策负担）
- 标记多个推荐项（消解推荐意义）
- 无依据推荐（必须能说出理由）

## 宿主无关原则

优先使用当前 Agent 宿主编辑器提供的结构化提问能力，而不是系统级弹窗。

调用优先级：
1. 当前宿主显式提供的结构化提问工具
2. 当前宿主的原生输入框 / Quick Pick / 交互式表单
3. 只有完全不可用时，退回普通聊天消息

不要把 Skill 写死成某一个编辑器的私有 API。由运行该 Skill 的模型根据自己所在环境选择对应工具。

## 示例

```
为什么问这个：前端框架决定了项目模板、组件库、状态管理等一系列后续选型。

你的项目前端框架倾向？

1. ⭐ Vue 3（推荐）— 团队已有 Vue 经验，渐进式学习曲线
2. React — 生态最庞大，社区资源丰富
3. Angular — 适合大型企业级应用，强规范
4. 其他 — 请直接输入你的想法或框架名称
```
